# Financial Engine End-to-End Walkthrough

This guide documents a concrete example in the **Vish Demo Lab** tenant using one initiative:

- **Initiative:** `Move tier-2 application support to India COE`
- **Initiative ID:** `fdb4e93d-a13c-4e1c-bb4f-d3381eba4052`
- **Purpose of example:** show how metric definitions, scenarios, benefit lines, cost lines, formula rows, bridge rows, attribute definitions, rollups, and front-end display relate to each other.

No credentials are included in this guide.

## 1. The scenario we configured

The hypothetical business case is:

> Move tier-2 application support into an India COE. The initiative creates revenue uplift by improving customer retention, creates margin benefit from that revenue uplift, avoids contractor spend, and incurs recurring run opex plus one-off implementation cost.

The walkthrough uses three months of 2026 data:

- January 2026
- February 2026
- March 2026

It uses three scenario lanes that align with the financial engine rollup keys:

- `plan_base` — Plan Base
- `plan_high` — Plan High / Upside
- `actual` — Actual

These scenario keys matter because the backend value bridge and portfolio rollups look for `plan_base`, `plan_high`, and `actual` when building base/high/actual cases.

## 2. Metric definitions

Metric definitions describe **what can be entered or calculated** in initiative financials.

For this example, four demo metric definitions were configured.

| Metric key | Label | Input or computed? | Benefit? | Formula / interpretation |
|---|---|---:|---:|---|
| `demo_revenue_impact` | Demo Revenue Impact | Input | No for net benefit rollup; used as a driver | Revenue uplift generated by the initiative. Used as a bridge row and as an input into the margin percentage formula. |
| `demo_margin_benefit` | Demo Gross Margin Benefit | Input | Yes, margin | Gross-margin dollar benefit. Included in benefits and net run-rate impact. |
| `demo_cost_avoidance` | Demo Cost Avoidance Benefit | Input | Yes, savings | Avoided recurring spend. Included in benefits and net run-rate impact. |
| `demo_margin_pct` | Demo Gross Margin % | Computed/read-only | No | `demo_margin_benefit / demo_revenue_impact * 100` |

### Key configuration pattern

An editable input metric is configured with:

```json
{
  "aggregation": "sum",
  "is_benefit": true,
  "benefit_class": "margin"
}
```

A computed/read-only metric is configured with:

```json
{
  "aggregation": "formula",
  "formula": "demo_margin_benefit / demo_revenue_impact * 100",
  "formula_inputs": ["demo_margin_benefit", "demo_revenue_impact"],
  "is_benefit": false
}
```

The important flag is:

```text
aggregation = formula
```

That is what makes the row calculated rather than user-entered.

## 3. Attribute definitions

Attribute definitions describe extra metadata fields that can be attached to financial lines.

The example has these attribute definitions:

| Attribute key | Label | Entity type | Value type | Options |
|---|---|---|---|---|
| `demo_benefit_owner` | Benefit Owner | `benefit_line` | `text` | none |
| `demo_realisation_confidence` | Realisation Confidence | `benefit_line` | `select` | High, Medium, Low |
| `demo_cost_vendor` | Cost Vendor / Team | `cost_line` | `text` | none |

### Current implementation note

Benefit-line attributes are persisted on each benefit line in the `attributes` object.

Example:

```json
{
  "demo_benefit_owner": "Transformation Finance",
  "demo_realisation_confidence": "High"
}
```

Cost-line attribute definitions can be registered, but the current `CostLineCreate` / `CostLineItem` API model does **not yet expose a cost-line `attributes` object**. So cost-line attribute definitions are available in the registry, but cost-line instance values are not currently captured by the cost-line API payload.

## 4. Benefit lines

Benefit lines describe the **business line item** beneath a metric definition.

In this example, three benefit lines were created:

| Benefit line | Metric definition | Impact type | Timing | Example attributes |
|---|---|---|---|---|
| Revenue uplift from support consolidation | `demo_revenue_impact` | recurring | Ramp starts Jan; full run-rate by Mar | Owner = Transformation Finance, Confidence = High |
| Gross margin conversion on revenue uplift | `demo_margin_benefit` | recurring | Calculated from revenue capture assumptions | Owner = Transformation Finance, Confidence = Medium |
| Avoided contractor spend | `demo_cost_avoidance` | recurring | Vendor support reduction | Owner = Transformation Finance, Confidence = High |

### How benefit lines are handled

A benefit line is not itself the monthly number. It is the named business line that monthly metric values attach to.

The monthly value rows reference:

- `metric_definition_id`
- `scenario_id`
- `benefit_line_id`
- `year`
- `month`
- `value`
- `status`

So the model is:

```text
Metric definition = what kind of number is this?
Benefit line = which named benefit is this number for?
Scenario = base/high/actual lane
Month = period
Value = amount
```

## 5. Cost lines

Cost lines are handled separately from benefit metric values.

The example uses two cost categories:

| Cost category key | Cost line name | Recurring? | Interpretation |
|---|---|---:|---|
| `demo_recurring_opex` | Demo recurring run opex | yes | Ongoing support/run cost. Subtracted from net run-rate impact. |
| `demo_implementation` | Demo one-off implementation | no | One-off implementation cost. Shown in bridge/cost totals but not subtracted from net run-rate impact. |

Each cost line contains both plan and actual amounts:

```json
{
  "name": "Demo recurring run opex",
  "category_key": "demo_recurring_opex",
  "year": 2026,
  "month": 1,
  "amount_plan": "20000",
  "amount_actual": "18000",
  "is_recurring": true
}
```

### How cost lines are handled

- `amount_plan` is used for plan/base/high calculations.
- `amount_actual` is used for actual calculations.
- `is_recurring = true` makes the cost part of recurring costs.
- `is_recurring = false` makes the cost part of one-off costs.
- Net run-rate impact subtracts recurring costs, not one-off costs.

## 7. Computed formula rows

The computed metric is:

```text
Demo Gross Margin % = Demo Gross Margin Benefit / Demo Revenue Impact * 100
```

Because every month uses a 40% gross-margin assumption, the computed formula row returns:

| Scenario | Jan | Feb | Mar |
|---|---:|---:|---:|
| Plan Base | 40.0000 | 40.0000 | 40.0000 |
| Plan High | 40.0000 | 40.0000 | 40.0000 |
| Actual | 40.0000 | 40.0000 | 40.0000 |

### How read-only formula rows are calculated

The frontend does **not** save formula metric values.

The backend calculates them at read time:

1. Load stored non-formula metric values.
2. Group values by tenant, initiative, scenario, year, and month.
3. Build a formula environment like:

   ```json
   {
     "demo_revenue_impact": 100000,
     "demo_margin_benefit": 40000
   }
   ```

4. Evaluate:

   ```text
   demo_margin_benefit / demo_revenue_impact * 100
   ```

5. Return a synthetic computed row with:

   ```json
   {
     "id": "formula:{initiative_id}:{scenario_id}:2026:1:{definition_id}",
     "value": "40.0000",
     "status": "approved",
     "note": "Computed formula metric"
   }
   ```

### How the frontend displays formula rows

In the initiative financial grid:

- Formula rows have ` - computed` appended to the label.
- Formula rows are `readOnly`.
- Formula cells get the `hot-computed-cell` CSS class.
- The user cannot edit them directly.
- If a formula value needs to change, the user edits the input rows feeding the formula.

Example visible row label:

```text
Demo Gross Margin % (Plan Base) - computed
```

## 8. Bridge row definitions

Bridge rows define how the value bridge should display rows.

The example has these configured bridge rows:

| Bridge row | Kind | Inputs | Sign | Calculation |
|---|---|---|---:|---|
| Demo Revenue Impact | `metric_set` | `demo_revenue_impact` | +1 | Sum the selected metric values. |
| Demo Gross Margin Benefit | `metric_set` | `demo_margin_benefit` | +1 | Sum the selected metric values. |
| Demo Cost Avoidance Benefit | `metric_set` | `demo_cost_avoidance` | +1 | Sum the selected metric values. |
| Demo Recurring Opex | `cost_set` | `demo_recurring_opex` | -1 | Sum matching cost lines and multiply by -1. |
| Demo Implementation Cost | `cost_set` | `demo_implementation` | -1 | Sum matching cost lines and multiply by -1. |
| Demo Net Run-rate Impact | `net` | automatic | +1 | Benefits total minus recurring costs. |

### Bridge row kind meanings

#### `metric_set`

A metric-set row sums all values for the listed metric definitions in the selected scenario.

Example:

```text
Demo Gross Margin Benefit, Plan Base
= 40,000 + 60,000 + 80,000
= 180,000
```

#### `cost_set`

A cost-set row sums matching cost categories.

If `sign = -1`, the displayed value is negative.

Example:

```text
Demo Recurring Opex, Plan Base
= (20,000 + 25,000 + 30,000) * -1
= -75,000
```

#### `net`

A net row is calculated by the backend as:

```text
benefits_total - recurring_costs
```

For this walkthrough, benefits total is:

```text
Gross margin benefit + Cost avoidance benefit
```

Revenue impact is displayed as its own bridge driver row, but is not counted as benefit in the net run-rate calculation.

1. **Metric Definitions**
   - Create or edit input metrics.

The dedicated financial endpoints returned the expected rollups. However, two broader dashboard/report endpoints showed partial inconsistency with the configurable financial engine data:

- `GET /dashboard`

This suggests the initiative financial tab, value bridge, and portfolio financial endpoints are aligned with the configurable engine, but some broader dashboard/reporting endpoints may still be using older or incomplete financial aggregation paths. ## 13. Mental model summary

Use this hierarchy:

```text
Financial engine configuration
  ├─ Metric definitions: what fields exist, input vs formula
  ├─ Scenario definitions: base/high/actual lanes
  ├─ Bridge rows: how values are displayed and summed
  └─ Attribute definitions: metadata fields available on lines

Initiative financial setup
  ├─ Benefit lines: named business benefits under metric definitions
  ├─ Metric values: scenario/month/value rows attached to benefit lines
  └─ Cost lines: plan/actual cost rows by category and recurring flag

Calculated output
  ├─ Formula rows: computed from input metric values, read-only
  ├─ Value bridge: configured metric/cost rows plus net row
  ├─ Portfolio financials: monthly/quarterly/yearly benefits, costs, net
  └─ Dashboards/reports: should consume rollups, but need final alignment check
```

The key end-user rule is:

> Users input benefit values and cost values. Formula rows are configured once in Admin and then calculated automatically in initiative financials. Bridge rows determine how the resulting values are grouped and interpreted in summaries.
