# One-Sentence Purpose

**Transmuter is a multi-tenant enterprise transformation SaaS platform that turns strategic initiatives into governed, financially validated, and realized business value.**

It replaces disconnected spreadsheets, slide decks, PMO trackers, and manually reconciled finance reports with one operating system for transformation value.

---

# The Problem

Enterprise transformations fail to create confidence because value management is fragmented.

- Strategy sits in board decks.
- Initiative detail sits in PMO spreadsheets.
- Finance validation sits in separate models.
- Delivery status sits in project tools.
- Benefits realization sits in monthly manual reconciliation.
- Shared platform, PMO, vendor, and change costs are often invisible or allocated late.

The result: executives cannot quickly answer, **"What value is bankable, what is realized, and what is at risk?"**

---

# Why Existing Tools Break

| Current alternative | What it does | Where it breaks |
|---|---|---|
| Spreadsheets | Flexible planning | Weak controls, version drift, no tenant governance |
| Work management tools | Tasks and milestones | Not finance-native, weak benefit validation |
| BI dashboards | Visual reporting | Usually downstream of stale manual data |
| FP&A / EPM tools | Planning and consolidation | Not initiative-operating systems |
| Consulting PMO decks | Executive narrative | Static, expensive, hard to sustain |

Transmuter connects the operating work to the financial value case and executive governance layer.

---

# The Solution

Transmuter provides a governed transformation control plane:

- Tenant onboarding, roles, and master data.
- Initiative lifecycle from idea to delivery.
- Financial engine for scenarios, metrics, baselines, formulas, and costs.
- Benefits Register and Bankable Plan governance.
- Benefit Tracking and Waterline realization views.
- Shared Costs allocation engine for fully loaded economics.
- Executive Dashboard and Control Tower reporting.
- PMO views for milestones, KPIs, risks, dependencies, and governance.
- AI assistant with tenant-aware context and human-in-the-loop write controls.

---

# Why Now

Transformation scrutiny is increasing while delivery complexity is rising.

- Boards expect evidence that transformation programs deliver measurable value.
- CFOs need defensible benefit validation, not optimistic initiative claims.
- CEOs need a simple value story linked to execution and risk.
- COOs need an operating cadence that exposes blockers before value slips.
- AI adoption increases the need for governance, traceability, and tenant isolation.
- Shared platforms and central transformation costs require allocation discipline.

The market needs transformation software that combines PMO execution with finance-grade value control.

# Finance Workflow

```text
Owner drafts benefit line
  -> transformation office reviews completeness
  -> Finance validates or rejects
  -> gate approval locks bankable plan
  -> actuals entered in benefit ledger
  -> Benefit Tracking compares actuals to locked baseline
  -> Waterline freezes workstream targets
  -> Control Tower reports direct and fully loaded economics
```

This creates a clear separation between planned value, validated value, bankable value, and realized value.

---

# Shared Costs Workflow

Use Shared Costs for central burdens that support multiple initiatives:

- PMO and benefits office.
- Platform and data foundation.
- Change management and training.
- Advisory, vendor, or implementation support.

Workflow:

1. Configure reporting treatment.
2. Create cost pools and periods.
3. Select allocation methods and targets.
4. Enter weights or fixed percentages where required.
5. Preview reconciliation.
6. Approve, lock, or void runs.
7. Review allocated burden in Control Tower.

---

- Equal split.

---

# Architecture Overview

```text
Angular 21 SPA
  <-> HTTPS / REST
FastAPI 0.115+ on Python 3.12
  <-> Supabase client with JWT-scoped access
Supabase PostgreSQL 15 + Auth + RLS
  <-> tenant-scoped data and policies
OpenRouter + PydanticAI agents
  <-> Langfuse traces and evals
Docker / Traefik / Hostinger deployment
```

The platform is implemented as a multi-tenant SaaS, not a single-tenant demo app.

# Security and Tenant Isolation

Core controls:

- Every tenant table uses `tenant_id`.
- Queries are tenant-scoped.
- Supabase RLS is mandatory on tenant tables.
- JWT carries user and tenant context.
- User-facing API calls avoid service-role data leakage patterns.
- RBAC differentiates platform admin, transformation office, initiative owner, and viewer.
- Money uses PostgreSQL `NUMERIC(15,4)`, Python `Decimal`, and string JSON responses.
- Raw PII must not be sent to external LLM APIs.
- Agent write actions require human-in-the-loop checkpoints.

# AI and Observability

AI is additive, not blocking core platform functionality.

Implemented design principles:

- PydanticAI agents with declarative specs.
- OpenRouter as the LLM gateway.
- Langfuse traces, evals, and prompt/version observability.
- Tenant-aware context.
- Suggested actions with confirmation before writes.
- Graceful degradation if LLM functionality is unavailable.

High-value AI use cases:

- Portfolio summarization.
- At-risk initiative discovery.
- Draft risks, milestones, KPI updates, cost lines, and financial entries.
- Executive narrative support.
